Overview

The goal of this notebook is doing some time-series analysis, end-to-end, from ETL to gathering insights.

The available data consists of two data-sets:

  • connectivity_events.csv: Comprised of device events (devices connection and disconnection to a server).

  • servers.csv: Comprised of server events (server creation and destruction).

Due to servers not being available all the time, "connection" and "disconnection" events not necessarily pair-up nicely (that is, for each "online" event there isn't necessarily a "offline" event). We will be using the server events data to fix these inconsistencies in the device events.

We can now use the rules below to add or remove events that will make the data consistent:

current event current server next event next server rule
online X online X Assume device was online from current event's timestamp until next event's timestamp
online X online Y Assume device was online from current event's timestamp until X's destruction time or next event's timestamp, whichever is smaller
online X offline X Normal case
online X offline Y Assume device was online from current event's timestamp until X's destruction time. Ignore next event.
offline X online X Normal case
offline X online Y Normal case
offline X offline X Ignore next event
offline X offline Y Ignore next event

The idea is using Panda's apply() function over the dataframe's rows to accomplish that:

In [28]:
import datetime

events_to_add = []
events_to_remove = []

def add_event(timestamp, user_id, device_id, connected, server_id):
    events_to_add.append({"timestamp": timestamp,
                          "user_id": user_id,
                          "device_id": device_id,
                          "connected": connected,
                          "server_id": server_id,
                          "event_id": None
                         })

def remove_event(event_id):
    events_to_remove.append(event_id)

def process_event_pairs(row):
    if row.device_id == row.next_device_id:
        if row.connected == True and row.next_connected == True:

            if row.server_id == row.next_server_id:
                # Assume device was online from current event's timestamp until next event's timestamp.
                add_event(row.timestamp,
                          row.user_id, row.device_id, False, row.server_id)

            else:
                # Assume device was online from current event's timestamp until X's destruction time or 
                # next event's timestamp, whichever is smaller
                add_event(min(row.server_destroyed_at, row.next_timestamp),
                          row.user_id, row.device_id, False, row.server_id)

        elif row.connected == True and row.next_connected == False:

            if row.server_id == row.next_server_id:
                # Normal case
                pass

            else:
                # Assume device was online from current event's timestamp until X's destruction time.
                # Ignore next event.
                add_event(row.server_destroyed_at,
                          row.user_id, row.device_id, False, row.server_id)

                remove_event(row.next_event_id)

        elif row.connected == False and row.next_connected == True:
            # Normal case
            pass

        elif row.connected == False and row.next_connected == False:
            # Ignore next event
            remove_event(row.next_event_id)
    
    else:
        # Last event for a device (does not pair-up)
        if row.connected == True:
            add_event(row.server_destroyed_at,
                      row.user_id, row.device_id, False, row.server_id)

import datetime

device_usage = []

def remove_time(datetime_object):
    return datetime.datetime.combine(datetime_object.date(), datetime.datetime.min.time())

def days_in_between(from_date, to_date):
    from_date = remove_time(from_date)
    to_date = remove_time(to_date)
    return [(from_date + datetime.timedelta(days=x)).date() for x in range((to_date - from_date).days + 1)]

def add_device_usage(user_id, device_id, from_datetime, to_datetime):
    for date in days_in_between(from_datetime, to_datetime):
        device_usage.append({"date": date,
                             "user_id": user_id,
                             "device_id": device_id})

def process_fixed_event_pairs(row):
    add_device_usage(row.user_id, row.device_id, row.timestamp, row.next_timestamp)

Thus, we may use forecasting methods which have the following characteristics:

  • It has a single variable.

  • It exibits a trend (power law growth).

  • It has a seasonal component.

From all the available forecast methods, there are four that apply to this behavior:

  • Autoregressive Integrated Moving Average (ARIMA)
  • Seasonal Autoregressive Integrated Moving-Average (SARIMA)
  • Seasonal Autoregressive Integrated Moving-Average with Exogenous Regressors (SARIMAX)
  • Holt Winters Exponential Smoothing (HWES)

ARIMA is suitable for a time series without a seasonal component, but in this particular case, the order of magnitude of the seasonal component is two orders of magnitude smaller then the trend component, thus using the method is viable.

Task 3 - Business insights

Large Majority of Users Own 1-2 Devices

A tiny share of users have more than 100 devices, thus, most of the revenue comes from users with a small number of devices (1-2 or 3-4). It seems like a lot of the users are still in the "prototype" phase of their projects (either that or they just don't require many devices to accomplish whatever they are trying to do).

It would make sense to provide more plans that are not based solely on the number of devices that are allowed.

It could be featured based or on usage (number of provisionings, periods of the day or week provisonings happen, etc).

Has a Weekly Seasonal Behavior

The number of devices decreases towards the weekend and reach their peak in the middle of the week. It seems that the device usage is linked to processes that are related to "corporate activities" (more intense during work days).

Assuming that the events (connections and disconnections) are solely related to provisioning, that makes sense, they should happen mostly over work days.

It might be interesting to get some info on the type of projects the users are involved though, so we could come up with new features that would be useful for this kind of user.

It would be interesting to understand why there are more events particularly in the middle of the week...

The obvious insight is that this seasonal behavior can be used to prepare the available infrastructure for short-term fluctuations in the number of connected devices.

Follows a Power Law Trend

The most obvious insight is that we can use this trend component to estimate how much the number of devices will grow long-term (and so the reveneu). This insight can be used to have the proper infrastructure in place for future growth.
